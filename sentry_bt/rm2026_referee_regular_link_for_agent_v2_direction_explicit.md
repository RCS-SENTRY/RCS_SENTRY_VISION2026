# RoboMaster 2026 V1.2.0 常规链路串口协议整理（面向代码修改 Agent）

> 来源：`RoboMaster 2026 机甲大师高校系列赛通信协议 V1.2.0（2026-02-09）`
> 范围：**常规链路**，即裁判系统服务器/主控模块经由**电源管理模块 User 串口**与机器人单片机通信的部分。
> 不包含：图传链路 `0x0302/0x0309/0x0310/0x0311`、雷达无线链路 `0x0A01~0x0A06`、自定义客户端协议第 2 章。
> 版本更新提醒：V1.2.0 明确修订了 `0x0105、0x0204、0x0209、0x020C、0x020D`，并删除 `0x0304`。

---


## 0. 读者约定（非常重要：区分收包和发包）

本整理稿统一使用以下方向标记，代码 Agent 必须严格按此理解：

- `[RX]`：**下位机/机器人单片机接收**。对应代码里的 **decode / unpack / parse / on_rx**。
- `[TX]`：**下位机/机器人单片机发送**。对应代码里的 **encode / pack / build / send**。
- `[FWD]`：协议里经过服务器/选手端/图传链路中转，但对当前单片机来说，仍需先判断自己是发送方还是接收方。
- 除非明确写了 `[TX]`，否则不要把该结构体拿去打包发送。
- 除非明确写了 `[RX]`，否则不要把该结构体拿去当接收包解析。
- 尤其注意：`0x0301` 是**机器人交互总包 cmd_id**，其内部不同 **子内容 ID** 方向可能不同，不能因为都挂在 `0x0301` 下面就默认全是接收，或默认全是发送。

### 0.1 本次对话里最容易混淆的一条
- `[TX] 0x0301 / sub_id = 0x0120`：**哨兵自主决策指令**。这是**机器人/单片机发出**，由服务器按规则处理。
- `[RX] 0x020D`：**哨兵自主决策信息同步**。这是**服务器下发给哨兵**的同步数据，不是 `0x0120`。

## 1. 物理链路与总帧格式

### 1.1 串口配置（常规链路）
- 波特率：`115200`
- 数据位：`8`
- 停止位：`1`
- 硬件流控：`无`
- 校验位：`无`

### 1.2 常规链路路径
常规链路由**裁判系统服务器**和**主控模块**进行数据转发，从**电源管理模块的 User 串口**收发数据。

### 1.3 时延与丢包（官方给出的参考值）
- 正常工作状态：延迟约 `130ms`，丢包率 `<1%`
- 网络环境较恶劣：延迟约 `200ms`，丢包率约 `3%`
- 官方说明：测量数据可能有误差，仅供参考

### 1.4 整包格式
```text
frame_header (5B) + cmd_id (2B) + data (nB) + frame_tail (2B CRC16)
```

#### frame_header 格式（5B）
```text
SOF(1) + data_length(2) + seq(1) + CRC8(1)
```

#### 字段说明
- `SOF`：1 字节，固定 `0xA5`
- `data_length`：2 字节，表示 **data 段长度**
- `seq`：1 字节，包序号
- `CRC8`：1 字节，帧头 CRC8

### 1.5 长度计算
- 整包总长度 = `5 + 2 + data_length + 2 = data_length + 9`

### 1.6 CRC
官方在附录给了完整示例代码：
- `CRC8_INIT = 0xFF`
- CRC8 生成多项式：`G(x)=x^8+x^5+x^4+1`
- `CRC16_INIT = 0xFFFF`
- CRC16 尾字节写入顺序：**低字节在前，高字节在后**

### 1.7 实现建议
建议解析器按下面顺序工作：
1. 扫描 `0xA5`
2. 读取 5 字节帧头并校验 CRC8
3. 取 `data_length`
4. 继续读取 `cmd_id + data + CRC16`，总长度为 `data_length + 4`
5. 校验整包 CRC16
6. 根据 `cmd_id` 分发

V1.2.0 相比旧版，常规链路里**明确修订**了以下命令：

### 3.2 明显需要在代码里做“版本兼容检查”的长度异常
文档里至少有两处**命令表长度**与**详细结构体**不一致，代码更新时要特别小心：
1. `0x0203`
   - 命令总表写：`16`
   - 详细表与结构体：`float x + float y + float angle = 12B`
2. `0x0208`
   - 命令总表写：`6`
   - 详细表与结构体：四个 `uint16_t`，实际 `8B`

这两处更可信的通常应是**详细表/结构体定义**，但如果接实机时长度不符，建议以实测包长为准并保留日志。

## 4.1 `0x0001` 比赛状态数据（11B，1Hz）

### 结构
```c
typedef _packed struct
{
    uint8_t  game_type : 4;
    uint8_t  game_progress : 4;
    uint16_t stage_remain_time;
    uint64_t SyncTimeStamp;
} game_status_t;
```

### 字段
- byte `0`
  - bit `0-3`：比赛类型
    - `1`：RoboMaster 机甲大师超级对抗赛
    - `2`：RoboMaster 机甲大师高校单项赛
    - `3`：ICRA RoboMaster 高校人工智能挑战赛
    - `4`：RoboMaster 高校联盟赛 3V3 对抗
    - `5`：RoboMaster 高校联盟赛步兵对抗
  - bit `4-7`：当前比赛阶段
    - `0`：未开始比赛
    - `1`：准备阶段
    - `2`：15 秒裁判系统自检阶段
    - `3`：5 秒倒计时
    - `4`：比赛中
    - `5`：比赛结算中
- byte `1-2`：`stage_remain_time`，单位 `秒`
- byte `3-10`：`SyncTimeStamp`，`UNIX 时间`，机器人正确连接裁判系统 NTP 服务器后生效

## 4.4 `0x0101` 场地事件数据（4B，1Hz）

```c
typedef _packed struct
{
    uint32_t event_data;
} event_data_t;
```

### 位定义
- bit `0`：己方与资源区**不重叠**的补给区占领状态，`1=已占领`
- bit `1`：己方与资源区**重叠**的补给区占领状态，`1=已占领`
- bit `2`：己方补给区占领状态，`1=已占领`（**仅 RMUL**）
- bit `3-4`：己方小能量机关状态
  - `0` 未激活
  - `1` 已激活
  - `2` 正在激活
- bit `5-6`：己方大能量机关状态
  - `0` 未激活
  - `1` 已激活
  - `2` 正在激活
- bit `7-8`：己方中央高地占领状态
  - `1` 被己方占领
  - `2` 被对方占领
- bit `9-10`：己方梯形高地占领状态，`1=已占领`
- bit `11-19`：对方飞镖最后一次击中己方前哨站或基地的时间，范围 `0~420`，开局 `0`
- bit `20-22`：对方飞镖最后一次击中己方前哨站或基地的目标
  - `0` 开局默认
  - `1` 前哨站
  - `2` 基地固定目标
  - `3` 基地随机固定目标
  - `4` 基地随机移动目标
  - `5` 基地末端移动目标
- bit `23-24`：中心增益点占领状态（**仅 RMUL**）
  - `0` 未占领
  - `1` 被己方占领
  - `2` 被对方占领
  - `3` 被双方占领
- bit `25-26`：己方堡垒增益点占领状态
  - `0` 未占领
  - `1` 被己方占领
  - `2` 被对方占领
  - `3` 被双方占领
- bit `27-28`：己方前哨站增益点占领状态
  - `0` 未占领
  - `1` 被己方占领
  - `2` 被对方占领
- bit `29`：己方基地增益点占领状态，`1=已占领`
- bit `30-31`：保留

## 4.5 `0x0104` 裁判警告数据（3B）

{
    uint8_t level;

    uint8_t count;

### 字段

## 4.9 `0x0203` 机器人位置数据（详表/结构体 12B，1Hz；命令总表写16B）

```c
typedef _packed struct
{
    float x;
    float y;
    float angle;

```

### 字段
- `0-3`：本机器人位置 `x`，单位 `m`
- `4-7`：本机器人位置 `y`，单位 `m`

### 重要备注
- 命令总表把 `0x0203` 的 data 长度写成了 `16`
- 但详细表和结构体只给出了 `3 * float = 12B`
- 这里属于**文档自相矛盾**项，更新代码时建议：
  1. 先按 **12B 结构体**实现
  2. 同时保留对 `16B` 包长的诊断日志
  3. 实机抓包确认最终包长

## 4.10 `0x0204` 机器人增益和底盘能量数据（8B，3Hz）【V1.2.0 修订项】

```c
typedef _packed struct
{
    uint8_t  recovery_buff;
    uint16_t cooling_buff;
    uint8_t  defence_buff;
    uint8_t  vulnerability_buff;
    uint16_t attack_buff;
    uint8_t  remaining_energy;
} buff_t;
```

### 字段与偏移
- offset `0`，1B：`recovery_buff`
  - 机器人回血增益，百分比
  - 例如值 `10` 表示**每秒恢复血量上限的 10%**
- offset `1`，2B：`cooling_buff`
  - 射击热量冷却增益的**直接值**
  - 值 `x` 表示热量冷却增加 `x/s`
- offset `3`，1B：`defence_buff`
  - 防御增益百分比
  - 值 `50` 表示 `50%`
- offset `4`，1B：`vulnerability_buff`
  - 负防御增益百分比
  - 值 `30` 表示 `-30%`
- offset `5`，2B：`attack_buff`
  - 攻击增益百分比
  - 值 `50` 表示 `50%`
- offset `7`，1B：`remaining_energy`
  - bit `0-6`：机器人剩余能量比例反馈
  - 仅在机器人剩余能量 `<50%` 时按位反馈；其余默认 `0x80`
  - 机器人初始能量视为 `100%`
  - bit 含义：
    - bit `0`：剩余能量 `>=125%`
    - bit `1`：剩余能量 `>=100%`
    - bit `2`：剩余能量 `>=50%`
    - bit `3`：剩余能量 `>=30%`
    - bit `4`：剩余能量 `>=15%`
    - bit `5`：剩余能量 `>=5%`
    - bit `6`：剩余能量 `>=1%`

## 4.11 `0x0206` 伤害状态数据（1B，伤害后发送）

```c
typedef _packed struct
{
    uint8_t armor_id : 4;
    uint8_t HP_deduction_reason : 4;
} hurt_data_t;
```

### 字段
- bit `0-3`：受伤模块 ID
  - 当扣血原因为**装甲模块被弹丸攻击、受撞击或离线**时，该值为装甲模块或测速模块 ID

## 5. `[FWD]` `0x0301` 机器人交互数据（总包，内部子内容方向各不相同）（118B，常规链路，上限30Hz）

这是常规链路里最容易写错的一个，因为它的 **cmd_id 固定是 `0x0301`**，但真正功能还要再看 data 里的 `data_cmd_id`（子内容 ID）。

### 5.1 统一外层结构
```c
typedef _packed struct
{
    uint16_t data_cmd_id;
    uint16_t sender_id;
    uint16_t receiver_id;
    uint8_t  user_data[x];
} robot_interaction_data_t;
```

### 5.2 规则约束
- 整个 `0x0301` 的**data 段长度固定 118B**
- 其中前 6B 是统一头
- 内容数据段 `user_data` 最大 `112B`
- 机器人交互总包长度不超过 `127B`
  - 因为要扣除 `frame_header(5) + cmd_id(2) + frame_tail(2) = 9B`
  - 再扣除数据段头 `6B`
- 每 `1000ms`
  - 英雄 / 工程 / 步兵 / 空中机器人 / 飞镖：接收上限 `3720B`
  - 雷达 / 哨兵：接收上限 `5120B`
- 整个 `0x0301` 上行频率上限 `30Hz`
- 仅限**己方通信**
- 若接收者为选手端，则仅可发给**发送者对应的选手端**

### 5.3 已开放的 `data_cmd_id`
| data_cmd_id | 内容长度 | 含义 |
|---|---:|---|
| `0x0200~0x02FF` | `x<=112` | 机器人之间通信 |
| `0x0100` | 2 | 选手端删除图层 |
| `0x0101` | 15 | 选手端绘制一个图形 |
| `0x0102` | 30 | 选手端绘制两个图形 |
| `0x0103` | 75 | 选手端绘制五个图形 |
| `0x0104` | 105 | 选手端绘制七个图形 |
| `0x0110` | 45 | 选手端绘制字符图形 |
| `0x0120` | 4 | 哨兵自主决策指令 |
| `0x0121` | 1+7=8 | 雷达自主决策指令 |

## 5.8 `0x0301` / 子内容 `0x0120`：哨兵自主决策指令（4B）

```c
typedef _packed struct
{
    uint32_t sentry_cmd;
} sentry_cmd_t;
```

### 位定义
- bit `0`：哨兵是否确认复活
  - `0`：确认不复活，即使复活读条已完成
  - `1`：确认复活，若复活读条完成则立即复活
- bit `1`：哨兵是否确认兑换立即复活
  - `0`：不兑换
  - `1`：确认兑换，若符合规则则立即消耗金币兑换
- bit `2-12`：哨兵将要兑换的发弹量值
  - 开局为 `0`
  - 修改后，哨兵在补血点即可兑换允许发弹量
  - **必须单调递增**，否则不合法
  - 例：从 `0 -> X`，表示消耗 `X` 金币成功兑换 `X` 允许发弹量；后续只能 `X -> X+Y`
- bit `13-16`：哨兵远程兑换发弹量的请求次数
  - 开局为 `0`
  - **必须单调递增且每次只能 +1**
- bit `17-20`：哨兵远程兑换血量的请求次数
  - 开局为 `0`
  - **必须单调递增且每次只能 +1**
- bit `21-22`：哨兵修改当前姿态指令
  - `1` 进攻姿态
  - `2` 防御姿态
  - `3` 移动姿态（默认）
- bit `23`：哨兵是否确认使能量机关进入“正在激活”状态，`1=确认`
- bit `24-31`：保留

### 官方处理顺序说明
服务器会按**从低位到高位**的顺序依次处理这些指令，直到全部成功或出现不能处理的指令为止。  
也就是说，如果低位指令已经因为金币/条件不满足失败，后续高位指令可能直接被忽略。

  - 该值必须**单调递增且每次仅 +1**
  - 若双倍易伤正在生效，则第二次请求会在第一次结束后生效
- byte `1-7`：密钥更新/验证指令
  - 每个字节均为 ASCII 字母或数字
  - 开局为随机值
  - `byte1` 为指令类型
  - `byte2-7` 为密钥值
  - 当 `byte1 == 1`：更新己方加密密钥
  - 当 `byte1 == 2`：将雷达破解的对方密钥上传服务器进行验证

### 限制
- 仅在**开局**和**每次对方破解成功使得己方加密等级提高时**可以修改密钥，其他时间修改无效
- 当 `byte1 == 2` 时，每次更新验证密钥后的 `10 秒内`，再次更新无效

## 6. 小地图交互（常规链路部分）

---

## 6.1 `0x0303` 选手端下发小地图数据（15B）

```c
typedef _packed struct
{
    float    target_position_x;
    float    target_position_y;
    uint8_t  cmd_keyboard;
    uint8_t  target_robot_id;
    uint16_t cmd_source;
} map_command_t;
```

### 基本规则
#### 云台手发送
- 命令码：`0x0303`
- 触发发送
- 两次发送间隔不得低于 `0.5s`

发送方式一：
1. 点击己方机器人头像
2. （可选）按键盘一个按键或点击对方机器人头像
3. 点击小地图任意位置

结果：
- 向己方选定机器人发送**地图坐标**
- 若点击对方机器人头像，则用**目标机器人 ID**代替坐标数据

发送方式二：
1. （可选）按键盘按键或点击对方机器人头像
2. 点击小地图任意位置

结果：
- 向己方**所有机器人**发送地图坐标
- 若点击对方机器人头像，则发送目标机器人 ID

#### 半自动控制机器人对应操作手发送
- 命令码：`0x0303`
- 触发发送
- 两次发送间隔不得低于 `3s`

方式：
1. （可选）按键盘按键或点击对方机器人头像
2. 点击小地图任意位置

结果：
- 向操作手对应机器人发送坐标
- 若点击对方机器人头像，则发送目标机器人 ID

### 额外说明
- 一台半自动控制机器人**既可接收云台手信息，也可接收对应操作手信息**
- 两种信息来源通过 `cmd_source` 区分

### 特殊重发机制（非常重要）
为降低机器人串口接收端偶发不稳定性带来的影响，`0x0303` 的服务器发送机制**不是只发一次**：

1. 选手端触发 1 次发送后  
2. 服务器会以 `100ms` 间隔**额外再发送 4 次**
3. 即：总共发 `5 次`
4. 此后直到下一次选手端再次触发发送之前，服务器还会以 `1Hz` 的频率持续发送**最近一次的包**
5. 触发时的连续发送和 1Hz 定频发送**彼此独立计时**

### 代码建议
- `0x0303` 必须做**重复包去重/幂等处理**
- 不要把 5 连发 + 1Hz 续发误判成新的用户指令

### 字段
- `0-3`：目标位置 `x`，单位 `m`
  - 若发送目标机器人 ID，则该值为 `0`
- `4-7`：目标位置 `y`，单位 `m`
  - 若发送目标机器人 ID，则该值为 `0`
- `8`：云台手按下的键盘通用键值
  - 无按键则为 `0`
- `9`：对方机器人 ID
  - 若发送的是坐标，则该值为 `0`
- `10-11`：信息来源 ID，具体 ID 对应见附录

## 6.3 `0x0307` 选手端接收路径数据（103B，上限1Hz）

```c
typedef _packed struct
{
    uint8_t  intention;
    uint16_t start_position_x;
    uint16_t start_position_y;
    int8_t   delta_x[49];
    int8_t   delta_y[49];
    uint16_t sender_id;
} map_data_t;
```

### 字段
- `0`：意图
  - `1`：到目标点攻击
  - `2`：到目标点防守
  - `3`：移动到目标点
- `1-2`：路径起点 `x`，单位 `dm`
- `3-4`：路径起点 `y`，单位 `dm`
- `5-53`：`delta_x[49]`，每个点相对于上一个点的 `x` 增量，单位 `dm`
- `54-102`：`delta_y[49]`，每个点相对于上一个点的 `y` 增量，单位 `dm`
- `103-104`：发送者 ID  
  > 注意：这里文档命令表写 data 长度 `103`，但结构体按字段相加是 `105B`。这也是一个潜在冲突点。  
  > 如果代码要做严格长度校验，建议实机抓包确认。

### 坐标系
- 小地图左下角为原点
- 向右为 `X` 正方向
- 向上为 `Y` 正方向
- 显示时会按场地尺寸和小地图尺寸**等比缩放**
- 超边界点显示在边界处

## 8. 给修改代码 Agent 的直接实现清单

### 8.1 解析器层
1. 串口设为 `115200 8N1`
2. 使用 `SOF=0xA5` 做帧同步
3. 按 `data_length + 9` 计算总包长
4. 校验 `CRC8(header)` 和 `CRC16(packet)`
5. 根据 `cmd_id` 分发
6. 对 `0x0303` 做重复包处理

### 8.2 数据结构层
1. 全部结构体使用 `packed`
2. 按官方字段更新以下修订项：
   - `0x0105`
   - `0x0204`
   - `0x0209`
   - `0x020C`
   - `0x020D`
3. 删除旧版 `0x0304` 相关代码
4. 对文档长度冲突项加入容错或日志：
   - `0x0203`
   - `0x0208`
   - `0x0307`

### 8.3 业务逻辑层
1. `0x0206` 仅是本地即时判伤，最终判定以服务器为准
2. `0x0301` 要同时检查：
   - `cmd_id == 0x0301`
   - `data_cmd_id`
   - `sender_id`
   - `receiver_id`
3. `0x0120/0x0121` 存在**单调递增**规则，不能随便覆盖发送
4. `0x0308` 文本是 `utf-16`，还要注意大小端

### 8.4 最值得加日志的地方
- 包长异常
- CRC 校验失败
- `0x0303` 重复包去重情况
- `0x0203/0x0208/0x0307` 文档长度冲突
- `0x0301` 的发送者 / 接收者 ID 非法
- `0x0120/0x0121` 的“非单调更新”非法请求

## 9. 文档内可见的冲突/风险点汇总（建议代码里留兼容）

1. **`0x0203` 长度冲突**
   - 命令总表：16
   - 详表/结构体：12

2. **`0x0208` 长度冲突**
   - 命令总表：6
   - 详表/结构体：8

3. **`0x0307` 长度冲突**
   - 命令总表：103
   - 结构体字段相加：105

4. **位域结构体不要直接跨编译器裸 memcpy 依赖**
   - 文档虽然给了 C 位域写法，但不同编译器/ABI 对位域布局可能不同
   - 更稳妥的方式：**按字节读取，再手动掩码解析**

5. **`0x0303` 不是一次触发只来一个包**
   - 有 5 次连续发送 + 1Hz 持续重发机制

6. **`0x0308` 文本编码是 UTF-16**
   - 显示中文没问题
   - 但发送时必须处理大小端

## 10. 最后给 Agent 的一句话总结

这是 RoboMaster 2026 V1.2.0 的**常规链路**协议。底层帧格式为 `A5 + data_length + seq + CRC8 + cmd_id + data + CRC16`，串口 `115200 8N1`。更新代码时优先修正 `0x0105/0x0204/0x0209/0x020C/0x020D`，删除 `0x0304`，并对 `0x0203/0x0208/0x0307` 的文档长度冲突做兼容和抓包核验。`0x0303` 需要去重，`0x0301` 需要二级分发到 `data_cmd_id`，`0x0120/0x0121` 需要遵守单调递增规则。
